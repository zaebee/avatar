## 1. Безопасность

*   **AES-256-GCM:** Алгоритм выбран верно, он обеспечивает и конфиденциальность, и целостность (AEAD).
*   **Web Crypto API:** Уязвимостей в самом API нет, но есть критические моменты реализации:
    *   **IV (Initialization Vector):** Вы ДОЛЖНЫ использовать уникальный IV (96 бит) для каждого шифрования. Повторное использование IV в GCM полностью компрометирует ключ. IV следует генерировать через `crypto.getRandomValues()` и прикреплять к зашифрованному сообщению (например, первые 12 байт).
    *   **Key Derivation:** Использовать "shared secret" напрямую как ключ — плохая практика. Рекомендуется использовать **PBKDF2** (с солью) для получения ключа из пароля. Соль также нужно передавать вместе с шифротекстом.
*   **Shared secret в env:** Для PoC допустимо. Для продакшена лучше использовать **Yandex Lockbox** для хранения секретов и передачи их в Cloud Function.
*   **Передача через email:** Основной риск — метаданные (тема письма, отправитель). Само тело письма защищено шифрованием. Учтите, что Base64 увеличивает размер данных на ~33%.

## 2. Архитектура

*   **IMAP-polling (5 мин):** Для условий деревни (где задержка в 5-10 минут не критична) — это отличное решение. Оно энергоэффективно для клиента (отправил письмо и забыл).
*   **Yandex MQ:** Правильный выбор. Очередь позволяет сглаживать пики и гарантирует, что пост не потеряется, если asi:one временно недоступен.
*   **API Gateway:** В данной схеме он **избыточен**. Cloud Function (IMAP-poller) может запускаться по триггеру (Timer/Cron) и напрямую писать в MQ. API Gateway обычно нужен для внешних HTTP-вызовов.

## 3. Надёжность

*   **asi:one недоступен:** MQ обеспечит хранение сообщений (Visibility Timeout). Обязательно настройте **Dead Letter Queue (DLQ)** для сообщений, которые не удалось обработать после N попыток, чтобы они не блокировали очередь.
*   **Ошибки дешифрования:** Если дешифровка не удалась (неверный ключ/битые данные), функцию нужно завершать с ошибкой, а сообщение из MQ (если оно уже там) отправлять в DLQ. Для PoC можно просто логировать в Cloud Logging.
*   **Фото > 10MB:** Почта Яндекса может ограничивать вложения, а Cloud Functions имеют лимиты на память.
    *   **Решение:** Обязательное сжатие на клиенте (через Canvas API в Web-encryptor) до ~1-2 MB перед шифрованием. Это сэкономит трафик в деревне и впишется в лимиты облака.

## 4. Стоимость

*   **Оценка 300 ₽/мес:** Вполне реалистична и даже может быть ниже.
    *   **Cloud Functions:** Первые 1 млн вызовов и 10 ГБ*час в месяц бесплатно. При запуске раз в 5 минут это всего ~8600 вызовов.
    *   **MQ:** Также имеет бесплатный уровень на 1 млн запросов.
    *   **Object Storage:** Основные затраты будут на хранение, но 1-2 ГБ фото стоят копейки.

## 5. Упрощения для PoC

*   **Что упростить:**
    1.  Убрать API Gateway.
    2.  Вместо полноценного архивирования писем (IMAP MOVE) на первом этапе можно просто помечать их как прочитанные (SEEN).
*   **Что оставить обязательно:**
    1.  Шифрование на клиенте (без него теряется смысл безопасности через публичную почту).
    2.  Очередь (MQ) — это фундамент надежности.
    3.  Object Storage — Cloud Functions не должны хранить файлы в себе.

---
**Вердикт:** Архитектура жизнеспособна и хорошо продумана для специфических условий (деревня + блокировки). Основное внимание уделите генерации IV и сжатию фото на стороне клиента.
